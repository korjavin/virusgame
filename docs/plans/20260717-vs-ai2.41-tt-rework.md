# vs-ai2.41: TT best-move + persistence across ID + bound fix

## Overview

Rework the transposition table in `backend/search/search.go` so it actually pays
off. Four defects (audit-confirmed, measured):

1. `tableEntry` stores scores only — no best move — so `orderedChildren` never
   benefits from prior search.
2. A fresh `searcher` (and fresh table) is built **inside** the iterative-deepening
   depth loop in both `Choose` (L125-137) and `chooseNodeBudget` (L62-73), so the
   table is thrown away every iteration → measured 0.0% cross-iteration benefit.
3. The probe gate `entry.depth >= depth && entry.ply == ply` returns hits as if
   they were exact, and the store side (`minimax`, L243-247) writes the value as
   exact **only when there was no beta cutoff** — but a node that completes its
   loop without ever raising alpha above the original alpha is a **fail-low**
   (upper bound), and it is currently stored and later returned as exact. Real
   correctness bug.

Fixes: add `bestAction` to `tableEntry`; probe the TT move first in
`orderedChildren`; hoist the table to one `searcher` per `Choose` /
`chooseNodeBudget` call (reused across depths); add exact/lower/upper bound flags
with standard fail-soft alpha-beta semantics (fixing the fail-low bug). Keep
node-budget modes bit-deterministic — no `range` over the map anywhere; the TT
move and stable sort keep move choice deterministic. `backend/search/incumbent/**`
is BYTE-FROZEN — do not touch it. Do NOT add 1-ply-eval move ordering (measured
worse).

Transposition potential is real: 34-46% of interior visits at depth>=4 are
same-position/same-ply repeats (3-action turns converge via different placement
orders). The classic win is TT-move ordering compounding across ID iterations.

## Technical Details

- Bound flag uniform rule (fail-soft, minimax alpha-beta with a shared window,
  NOT negamax — alpha/beta are not negated between plies): after searching a node
  with entry window `[alphaOrig, betaOrig]`, `best <= alphaOrig` is a fail-low
  (upper bound), `best >= betaOrig` is a fail-high (lower bound), otherwise exact.
  This is identical for maximizing and minimizing nodes and subsumes the cutoff
  case, so the separate `cut` bool is no longer needed for the store.
- Determinism: point lookups `s.table[key]` are deterministic; `bestAction` is a
  deterministic function of prior search; `sort.SliceStable` on integer orders is
  deterministic. Never `range` the map.
- Cross-iteration value reuse is naturally blocked by `entry.depth >= depth`
  (within one ID iteration every node at ply p has the same remaining depth, so a
  later ID iteration's deeper requirement fails the gate) — the compounding win is
  TT-**move** ordering carried forward, not stale value cutoffs. This is correct
  and intended.
