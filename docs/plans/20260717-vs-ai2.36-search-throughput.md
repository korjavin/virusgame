# vs-ai2.36: Search throughput — Position path + connectivity dedup + budget

## Overview

Amplify the proven space-race eval (vs-ai2.34) by making the production search
generate more nodes per move. Profiling (bd `vs-ai2-throughput-movegen-not-eval`)
shows move generation is 78.5% of search CPU; the leaf evaluator is a non-lever.
PR #82 (child bead vs-ai2.36.1) already converted `game.State` hot accessors to
pointer receivers (~3.2x on `Apply`). This bead lands the remaining, higher-payoff
work, each item measured SEPARATELY against the gate battery:

1. Route the production search through the existing (unused) `game.Position` path
   (`NewPosition` + `ForEachSearchAction` + `ApplySearch`) instead of
   `State.LegalActions()` + `State.Apply()`. `ForEachSearchAction` PRUNES
   strategically-interchangeable neutral pairs → 1.5-3.4x fewer children. This is
   a strength-relevant behavioral change on boards where the branch set exceeds 32
   (12x12+); small boards (5x5/6x6) stay EXACT by design and their pinned goldens
   must not move.
2. Dedup the connectivity floodfill that is provably safe: `NewPosition` computes
   `State.connected(current)` twice on the identical pre-move board (once in
   `moveTargets`, once in `strategicNeutralPairs`). Only dedup what is provably
   safe on the SAME board+player — do NOT reuse across the pre/post-move mutation
   boundary (that is the part vs-ai2.36.1 honestly skipped and it stays skipped
   unless provably safe).
3. Raise `search.ProductionBudget` 600ms → 1000ms (owner pre-authorized up to 1s).
   Incumbent budget stays frozen at its own value.

Combined, this should lift completed-turn depth on dense 12x12/16x16/20x20 boards,
where the space-race eval's edge is currently node-starved (bead scale probe).

## Technical Details

- `ForEachSearchAction` keeps positions with `len(moves)+C(owned,2) <= 32` UNPRUNED,
  so 5x5/6x6 branch sets (and their pinned goldens) are identical. Pruning only bites
  on 12x12+, where no golden pins exact nodes — the strangler/differential gates arbiter.
- `ApplySearch` uses `State.applyGenerated` (skips the legality BFS) and shares
  mutation/elimination/terminal/advance semantics with `Apply`; the Position↔State
  equivalence tests already guard this contract.
- The search is anytime and always saturates its budget; raising the budget adds
  depth, it does not change determinism of the node-budget gates (those use
  `ChooseNodeBudget`, wall-clock-free).
