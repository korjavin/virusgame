# vs-ai2.35: Strangler-aware search (opponent ordering, threat extensions, root safety filter)

## Overview

Make the SEARCH see squeezes, complementing the vs-ai2.34 anti-strangulation
eval. Three cheap, independent, individually-toggleable search-side levers in
`backend/search/search.go`. Each is measured separately on the strangler gate;
a lever that does not measurably improve any gate is disabled/reverted, not
shipped. This plan produces the code + deterministic unit tests + a measurement
harness; the win-rate measurement itself is run out-of-band (env-gated, slow).

Key constraints (hard):
- Edit ONLY `backend/search/search.go` (+ its test) and add ONE new arena test
  file `backend/arena/lever_sweep_test.go`.
- `backend/search/evaluate.go` is READ-ONLY (an SPSA run owns it).
- `backend/search/incumbent/` is BYTE-FROZEN — never touch.
- Deterministic modes (`ChooseDepth`, `ChooseNodeBudget`) stay byte-deterministic.
- The golden `TestSearchMatchesOriginMainAtFixedDepthAndNodes` will shift; re-pin
  it HONESTLY from what the code actually produces (defaults all-on).

## Context (from discovery)

- `orderedChildren(state, ttMove, hasTT)` (search.go ~L402) builds and sorts the
  child list by an integer `order` (descending). Lever 1 adds a term here.
- `minimax(state, depth, alpha, beta, ply)` (search.go ~L199) is the 1v1 PVS
  recursion. Lever 2 adds a bounded depth extension here.
- `atDepth(state, depth)` (search.go ~L157) builds root children via
  `orderedChildren` then `preservingChildren`. Lever 3 filters candidates here.
- `s.root` (root player), `s.multi` (>2 active players) are on `searcher`.
- `preservingFallback`/`preservingChildren` are the "keep a legal, non-self-
  eliminating action" (owner) motif — Lever 3 must preserve at least one.
- `evaluate.go` defines the `SetEvalParams` tuner-only global-injection motif;
  mirror it for the lever toggles.
- `child` struct has `action`, `state`, `order`. Lever 2 needs a per-edge
  "threat" flag; add a field to `child`.
- Golden pins live in `search_test.go` `TestSearchMatchesOriginMainAtFixedDepthAndNodes`.
- Arena helpers reused by the harness: `stranglerOpenings`, `playBalancedOpenings`,
  `Wilson95`, `MobilityAttacker`, `Instrument`, `TelemetryNodeBudget` (all in
  `backend/arena`).

## Technical Details

- `strangleCount`: iterate the 8 neighbors of `target` within board bounds; count
  cells whose `Owner == root` and `Kind` is `Normal` or `Base`. No allocation.
- Lever 1 weight `*1000`: max 8 neighbors → ≤ 8000, strictly below the capture
  tier (10_000) and above retain-turn (100). Heuristic constant owned by search.
- Lever 2 extension: `ext` starts at 2 at every top-level `minimax` call from
  `atDepth`; decremented only when an extension is actually taken. Extending
  means recursing at `depth` instead of `depth-1`, so a threat line searches one
  ply deeper, capped twice per path. `running()`/node-limit checks unchanged, so
  `ChooseNodeBudget` still terminates deterministically.
- Lever 3 floor: "preserving-action count" = number of legal actions on the
  post-reply state that keep `s.root` active (reuse the `preservingFallback`
  active-after-apply idea). K=3 caps the opponent-reply scan.
