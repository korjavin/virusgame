# vs-ai2.44: leader-aware multiplayer aggression (eval)

## Overview

Production eval is opponent-agnostic. In 3-4 player games it fights the nearest
opponent while the leader grows, gets ganged up, and is eliminated first
(baseline vs-ai2.43: 4p prod vs aggressive heuristic field wins 5% at 25% fair
share, CI entirely below fair; bottom-two placement 16/20). Add three
**env-tunable, default-off** eval levers so an offline multiplayer-ladder sweep
can pick the winning constants. This task ships the instrumented levers + tests;
a follow-up commit (done by the human executor after the sweep) bakes the winning
constants and deletes the env hooks.

**Hard invariant: with all knobs at their default (0/off) the eval output is
byte-identical to today's, AND for any knob value the eval is byte-identical
whenever `active <= 2` (1v1, or a 4p game collapsed to two survivors).** All
three levers are gated behind `active > 2`, so 1v1 is untouched by construction —
this is what protects the 1v1 no-regression gates.

## Testing Strategy

- Unit tests in `backend/search/evaluate_test.go`:
  - default-off no-op: all knobs 0 → `evaluateAll` equals a captured baseline on a
    fixed 3p and 4p position.
  - 1v1 invariance: for a 2-player position, setting every knob to a large value leaves
    `evaluateAll` byte-identical (the `active > 2` gate).
  - directional: in a crafted 3p/4p position, enabling each lever moves the intended

### Task 3: Lever 2 — base-proximity aggression toward the leading opponent
- [ ] after the cut-bonus loop, add a term gated by `active > 2 && baseProxGain != 0`:

## Technical Details

- All per-mille integer math; clamp `mult >= 100`; `max(1, total)` guards divide-by-zero.
- `stand`/`share`/`fairShare` computed once per eval, only when a lever is live and `active > 2`.
- No change to the maxN differential (`utility[p] = raw[p] - avg(others)`) — levers act through `raw`.
