# vs-ai2.42: maxN shallow pruning + 1v1 PVS

## Overview

Add the two biggest cheap depth levers to `backend/search/search.go`, on top of
the vs-ai2.41 TT rework (bestAction + fail-soft bounds + cross-ID persistence).
Both are EXACT (same chosen move at a fixed depth as today) — they only cut node
count, which at a fixed node budget buys deeper search.

1. **maxN immediate/shallow pruning (multiplayer — currently ZERO pruning).**
   `maxN` (L276-317) expands every child at every node. The mover picks the child
   maximizing its own `values[player-1]`. The maximum any child can return for the
   mover is an immediate win at the child, `mateScore - (ply+1)` (terminal winner
   score; heuristic and deeper-win scores are strictly smaller). Once a child hits
   that ceiling, no sibling can beat it → break. This is the classic *immediate
   pruning* for maxN: safe with NO assumption about score sums (this eval is not
   constant-sum), fully deterministic, zero effect on 1v1 (maxN unused there).
   Move ordering already sorts a winning child first (`+1_000_000` bonus), so the
   cut fires on the first child when a winning move exists.

2. **1v1 PVS in `minimax` + the root `atDepth` 1v1 branch.** First child searched
   full-window `[alpha,beta]`; every later child searched null-window (maximizing:
   `[alpha,alpha+1]`; minimizing: `[beta-1,beta]`); on a fail that lands strictly
   inside the window, re-search full-window. The vs-ai2.41 TT move seeds child 0,
   which is exactly the precondition where PVS pays (strong first move ⇒ most
   null-window scouts fail low and cost one node). Reference pattern lives on
   branch `codex/vs-ai2.31-parallel-search` (search.go null-window/re-search loop);
   implement fresh on current search.go.

Aspiration windows: SKIPPED (optional, adds root re-search/widen complexity and a
harder-to-pin golden for marginal gain over TT-seeded PVS). Note in PR; add later
if measured worthwhile.

## Technical Details

- **Why immediate pruning is safe here:** maxN generalizes minimax; the mover
  maximizes its own component independently. The only universally-valid upper
  bound on that component without a constant-sum assumption is the terminal win
  `mateScore - (ply+1)` reachable at the shallowest child. Cutting when a child
  meets it discards siblings that provably cannot exceed it. No deep pruning (that
  would need a sum bound this non-constant-sum eval does not provide).
- **PVS side asymmetry:** this is min/max alpha-beta with a shared window, not
  negamax. Maximizing scout window is `[alpha, alpha+1]` (probe "can any sibling
  beat alpha?"); minimizing scout window is `[beta-1, beta]`. Re-search only on the
  strictly-inside-window result, matching the reference on
  `codex/vs-ai2.31-parallel-search`.
- **Determinism:** no randomness, no wall-clock in node-budget mode; PVS/pruning
  are pure functions of state ordering (stable sort + TT move). Re-pinned goldens
  are stable across runs. No new map iteration.
