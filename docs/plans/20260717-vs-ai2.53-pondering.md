# vs-ai2.53: permanent-brain pondering

## Overview
The production bot idles while the human thinks (5-30 s/turn = 5-30x its 1 s move
budget, free compute). Implement **permanent-brain pondering**: when it is the
opponent's turn, run an open-ended search from the current position (opponent to
move) that fills the SAME transposition table the next real search will reuse.
Cancel/restart on every authoritative position update; when it becomes our turn,
the real `Choose` starts warm and reaches materially deeper.

Key benefit mechanism: the shared TT's stored `bestAction` warms move ordering
for the real search (alpha-beta cutoffs → deeper per node). Score-cutoff reuse is
ply-gated (search.go L213/L311) and therefore stays sound across the ponder→real
ply offset — no correctness risk.

## Context (from discovery)
- Files/components involved:
  - `backend/search/search.go` — TT is created per `newSearcher` (fresh per
    `Choose` call). Add a `Session` type owning a persistent table; `Choose`,
    `ChooseDepth`, `ChooseNodeBudget` stay untouched (deterministic gates + goldens).
  - `backend/cmd/bot-hoster/bot_client.go` — `startSearch` (L545), the
    context/positionVersion/cancel machinery (L522-568), `calculateAndQueueAction`
    (L570). Ponder is scheduled from the existing `startSearch` call sites when it
    is NOT our turn.
  - `backend/cmd/bot-hoster/bot_search_test.go` — existing concurrency tests +
    `testBot` helper.
- Related patterns found: `context.WithTimeout(ProductionBudget)` + `searchCancel`
  cancel-on-update; `searchVersion == positionVersion` dedup; version-guarded send.
- Constraints:
  - **Determinism gates unaffected**: `ChooseDepth`/`ChooseNodeBudget` and package
    `Choose` are NOT modified. Golden test `TestSearchMatchesOriginMainAtFixedDepthAndNodes`
    must stay green (prove).
  - **TT sharing race-safe**: ponder and real search never run concurrently. A
    single `searchMu` mutex is held across each search invocation, so at most one
    goroutine ever touches the shared session table (single-writer by construction;
    cancellation just makes the previous holder release promptly).
  - **Memory**: fresh session per game + an in-searcher size cap that clears the
    table in place on overflow (a cache, not state — correctness unaffected).

## Technical Details
- `Session.run` and package `Choose` share the same `atDepth`/`minimax`/`maxN`
  engine; the only difference is table lifetime + the (cap-guarded) `store`.
- Lock order is always `searchMu` → `b.mu` (never the reverse), so no inversion:
  `startSearch` takes `b.mu` and spawns; `calculateAndQueueAction` takes `searchMu`
  then briefly `b.mu` for the send guard.
- ponytail ceiling: pondering only (re)starts from `startSearch` call sites
  (`startGame`, `handleGameState`/`turn_change`), so it covers the opponent's first
  (usually longest) think but pauses across the opponent's later sub-moves until the
  next `game_state`. Extending restart to the sub-move handlers is a follow-up if
  measured to matter.
